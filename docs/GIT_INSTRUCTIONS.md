# Git для монорепозитория (subtree)

## Правила работы с монорепозиторием

### Структура репозитория

```
RoomEditor/
├── backend/     # Бэкенд (из roomEditorBackend)
├── frontend/    # Фронтенд (из roomEditorFrontend)
├── docs/        # Документация
── README.md
```

---

### 1. Начало работы с веткой

Начинаю работу в отдельной ветке. Для этого переключаюсь на основную ветку:
```bash
git checkout main
```

Подтягиваю изменения из основной ветки:
```bash
git pull origin main
```

Создаю новую ветку, в которой буду вести разработку:
```bash
git checkout -b название_новой_ветки
```

---

### 2. Внесение изменений в кодовую базу

Вношу изменения в файлы (пишу код), потом когда хочу залить — проверяю, что git видит мои изменения:
```bash
git status
```

Эта команда выведет **красные** и **зелёные** файлы:
- **Зелёные** — файлы, которые git сможет закоммитить
- **Красные** — git не может закоммитить

Чтобы файлы стали зелёными, нужно написать:
```bash
git add путь_до_папки_или_файла
```

---

### 3. Коммит изменений

Когда все мои файлы зелёные, я могу сделать коммит:
```bash
git commit -m "комментарий к моим изменениям"
```

---

### 4. Отправка изменений в GitHub

Чтобы залить изменения в GitHub, пишу:
```bash
git push origin название_новой_ветки
```

---

### 5. Создание Pull Request

Теперь я могу сделать pull request. Захожу для этого на GitHub в свой проект, там сверху есть кнопка **pull request**. Нажимаю, попадаю на страницу pull request.

**Pull request** — это механизм GitHub для упрощённого процесса слияния. Он объединяет несколько веток в одну (merge), то есть мою новую ветку с основной.

---

### 6. Обновление subtree из оригинальных репозиториев

Если в оригинальных репозиториях появились изменения, их можно подтянуть:

**Обновить бэкенд:**
```bash
git subtree pull --prefix=backend backend main --squash
```

**Обновить фронтенд:**
```bash
git subtree pull --prefix=frontend frontend main --squash
```

> `--squash` объединяет все изменения в один коммит. Без него — каждый коммит будет отдельным.

---

### 7. Отправка изменений обратно в оригинальные репозитории

Если я внёс изменения в папку `backend/` или `frontend/` и хочу отправить их обратно в оригинальные репозитории:

**Отправить изменения бэкенда:**
```bash
git subtree push --prefix=backend backend main
```

**Отправить изменения фронтенда:**
```bash
git subtree push --prefix=frontend frontend main
```

---

### 8. Полезные команды

| Команда | Описание |
|---|---|
| `git log --oneline --graph` | Показать историю коммитов в виде графа |
| `git log --oneline --graph --all` | Показать все ветки в виде графа |
| `git remote -v` | Показать список удалённых репозиториев |
| `git branch -a` | Показать все ветки (локальные и удалённые) |
| `git status` | Показать статус рабочих файлов |

---

### 9. Удаление remote'ов (если больше не нужны)

```bash
git remote remove backend
git remote remove frontend
```

> ⚠️ После удаления remote'ов команды `subtree pull` и `subtree push` перестанут работать.

---

### 10. Важные правила

1. **Всегда работай в отдельной ветке** — не коммить напрямую в `main`
2. **Перед началом работы** — подтяни последние изменения из `main`
3. **Перед пушем** — проверь `git status`, что все нужные файлы добавлены
4. **Для subtree** — указывай правильный `--prefix` (папку) при pull/push
5. **Комментарии к коммитам** — пиши понятно, что именно изменилось
